# Паттерны микросервисов

> Ключевые паттерны для построения распределённых систем. Примеры на PHP 8.x и в псевдокоде.

Микросервисная архитектура разбивает приложение на independently развёртываемые сервисы. Это решает проблемы масштабирования и автономии команд, но добавляет сложность распределённых систем: сетевые сбои, согласованность данных, наблюдаемость. Паттерны ниже — типовые решения этих проблем.

## Коммуникация

### API Gateway

**Проблема:** клиент (браузер, мобильное приложение) не должен знать о десятках внутренних сервисов, их адресах и протоколах. Прямые вызовы создают связанность и проблемы с CORS, auth, версионированием.

**Решение:** единая точка входа, которая маршрутизирует запросы к нужным сервисам и берёт на себя сквозные задачи.

Что обычно делает gateway:
- маршрутизация и композиция запросов (агрегация ответов нескольких сервисов);
- аутентификация и авторизация (проверка JWT, передача identity дальше);
- rate limiting, кэширование, логирование, метрики;
- терминация TLS, трансформация протоколов (REST ↔ gRPC).

```
                ┌──────────────┐
   Клиент ──▶   │  API Gateway │ ──┬──▶ Auth Service
                └──────────────┘   ├──▶ Order Service
                                   ├──▶ Catalog Service
                                   └──▶ Payment Service
```

**BFF (Backend for Frontend)** — вариант gateway, где для каждого типа клиента (web, mobile) свой шлюз с оптимизированными под него ответами.

> Готовые решения: Kong, Traefik, AWS API Gateway, Nginx, Symfony Gateway. Риск — gateway становится «узким местом» и единой точкой отказа, поэтому его делают stateless и масштабируют горизонтально.

### Sync / Async messaging

Два стиля межсервисного взаимодействия:

| | Синхронное (Sync) | Асинхронное (Async) |
|---|-------------------|---------------------|
| Протоколы | HTTP/REST, gRPC | очереди сообщений (RabbitMQ, Kafka, SQS) |
| Поведение | запрос ждёт ответа (блокировка) | «отправил и забыл», ответ позже/никогда |
| Связанность | сильная (temporal coupling) | слабая |
| Отказоустойчивость | падение получателя ломает вызов | сообщение ждёт в очереди |
| Когда применять | нужен немедленный ответ (запрос данных) | фоновая обработка, события, нагрузка |

```php
<?php
// Синхронный вызов (PSR-18 HTTP-клиент)
$response = $httpClient->sendRequest(
    $requestFactory->createRequest('GET', 'http://catalog/products/42')
);
$product = json_decode($response->getBody()->getContents(), true);

// Асинхронная публикация в очередь (псевдокод, напр. через symfony/messenger)
$messageBus->dispatch(new OrderPlaced(orderId: 42));
// сервис не ждёт — обработчик в другом сервисе подхватит событие
```

> Правило: для **команд/запросов**, где нужен ответ сейчас — sync; для **уведомлений о событиях** и развязки сервисов — async. Чрезмерное увлечение sync-вызовами приводит к «распределённому монолиту».

### Event-Driven Architecture (EDA)

**Проблема:** при синхронной связи сервис-источник должен знать всех потребителей и ждать их. Это плохо масштабируется и создаёт каскадные отказы.

**Решение:** сервисы общаются через **события** — факты о том, что что-то произошло (`OrderPlaced`, `PaymentReceived`). Источник публикует событие в брокер и не знает, кто его обработает. Потребители подписываются независимо.

```php
<?php
// Сервис заказов публикует факт
final class OrderPlaced
{
    public function __construct(
        public readonly int $orderId,
        public readonly int $userId,
        public readonly float $total,
        public readonly string $occurredAt,
    ) {}
}

// Несколько сервисов независимо реагируют на одно событие:
// - Payment Service  → инициирует оплату
// - Notification      → шлёт письмо
// - Analytics         → пишет метрику
// - Inventory         → резервирует товар
```

Плюсы: слабая связанность, лёгкое добавление новых потребителей, естественная масштабируемость.
Минусы: сложнее отлаживать (нет линейного стека вызовов), eventual consistency, нужна идемпотентность потребителей.

**Виды событий:**
- **Event Notification** — тонкое уведомление («заказ изменился», за деталями сходи в API).
- **Event-Carried State Transfer** — событие несёт все данные (потребителю не нужно дозапрашивать).

## Устойчивость (Resilience)

### Timeouts & Retries with jitter

**Проблема:** сетевой вызов может зависнуть навсегда (без таймаута) или временно упасть. Простые повторы синхронно всеми клиентами создают «retry storm» — лавину запросов, добивающую сервис.

**Решение:** ограничить время ожидания (timeout) + повторять с **экспоненциальной задержкой и случайным разбросом (jitter)**.

```php
<?php
function callWithRetry(callable $operation, int $maxAttempts = 3): mixed
{
    $baseDelayMs = 100;

    for ($attempt = 1; $attempt <= $maxAttempts; $attempt++) {
        try {
            return $operation();          // у самой операции должен быть timeout!
        } catch (\Throwable $e) {
            if ($attempt === $maxAttempts || !isRetryable($e)) {
                throw $e;
            }
            // Экспоненциальная задержка: 100, 200, 400 мс...
            $expDelay = $baseDelayMs * (2 ** ($attempt - 1));
            // Full jitter: случайное значение от 0 до expDelay
            $delayMs = random_int(0, $expDelay);
            usleep($delayMs * 1000);
        }
    }
}

function isRetryable(\Throwable $e): bool
{
    // Повторять только временные сбои: 5xx, таймауты, разрыв соединения.
    // НЕ повторять 4xx (например, 400/404) — это не починится повтором.
    return $e instanceof NetworkException || $e instanceof ServerErrorException;
}
```

Ключевые моменты:
- **Timeout обязателен** — иначе один зависший вызов блокирует поток/ресурсы.
- **Jitter** размазывает повторы во времени, предотвращая синхронные всплески.
- Повторять только **идемпотентные** и **временные** ошибки (см. Idempotency).
- Комбинируется с Circuit Breaker, чтобы не повторять в заведомо мёртвый сервис.

### Circuit Breaker

**Проблема:** если сервис «лежит», продолжать слать в него запросы бессмысленно — это тратит ресурсы и замедляет вызывающего (каждый запрос ждёт таймаута). Сбой может каскадно распространиться.

**Решение:** «предохранитель» отслеживает ошибки и при превышении порога **размыкает цепь** — запросы сразу фейлятся, не доходя до больного сервиса. Периодически пропускает пробный запрос для проверки восстановления.

Три состояния:
- **Closed** (замкнут) — запросы идут нормально; считаются ошибки.
- **Open** (разомкнут) — порог ошибок превышен; запросы мгновенно отклоняются (срабатывает fallback).
- **Half-Open** (полуоткрыт) — по таймеру пропускается пробный запрос: успех → Closed, ошибка → снова Open.

```
   Closed ──(ошибок > порога)──▶ Open
     ▲                            │
     │                     (таймаут истёк)
     │                            ▼
     └──(пробный успех)──── Half-Open ──(пробный фейл)──▶ Open
```

```php
<?php
class CircuitBreaker
{
    private int $failures = 0;
    private string $state = 'closed';
    private ?int $openedAt = null;

    public function __construct(
        private int $threshold = 5,      // ошибок до размыкания
        private int $cooldown = 30,      // секунд в Open перед пробой
    ) {}

    public function call(callable $operation): mixed
    {
        if ($this->state === 'open') {
            if (time() - $this->openedAt < $this->cooldown) {
                throw new CircuitOpenException();  // не дёргаем больной сервис
            }
            $this->state = 'half-open';            // пора попробовать
        }

        try {
            $result = $operation();
            $this->reset();                        // успех — закрываем
            return $result;
        } catch (\Throwable $e) {
            $this->recordFailure();
            throw $e;
        }
    }

    private function recordFailure(): void
    {
        $this->failures++;
        if ($this->failures >= $this->threshold || $this->state === 'half-open') {
            $this->state = 'open';
            $this->openedAt = time();
        }
    }

    private function reset(): void
    {
        $this->failures = 0;
        $this->state = 'closed';
        $this->openedAt = null;
    }
}
```

> В реальных системах состояние хранят в Redis (общее для всех инстансов). Библиотеки: Resilience4j (Java), Polly (.NET), ackintosh/ganesha (PHP).

### Fallback

**Проблема:** когда вызов всё-таки провалился (circuit open, таймаут, ошибка), нужно деградировать **gracefully**, а не показывать пользователю ошибку.

**Решение:** запасной вариант ответа при недоступности основного источника.

Стратегии fallback:
- вернуть **кэшированное** (возможно, устаревшее) значение;
- вернуть **значение по умолчанию** или пустой результат;
- вызвать **альтернативный сервис**;
- поставить задачу в очередь на потом (для записи).

```php
<?php
function getRecommendations(int $userId): array
{
    try {
        return $this->circuitBreaker->call(
            fn() => $this->recommendationService->fetch($userId)
        );
    } catch (\Throwable $e) {
        // Деградация: не персональные, а популярные товары из кэша
        $this->logger->warning('Recommendations fallback', ['user' => $userId]);
        return $this->cache->get('popular_products', fn() => []);
    }
}
```

> Принцип graceful degradation: лучше показать «общие рекомендации», чем сломать всю страницу. Fallback почти всегда идёт в паре с Circuit Breaker.

### Rate Limiting

**Проблема:** нужно защитить сервис от перегрузки, злоупотреблений (DDoS, скрейпинг) и обеспечить справедливое распределение ресурсов между клиентами.

**Решение:** ограничение числа запросов от клиента за единицу времени.

Алгоритмы:

| Алгоритм | Суть |
|----------|------|
| **Token Bucket** | ведро токенов пополняется с фиксированной скоростью; запрос берёт токен; нет токенов → отказ. Допускает всплески. |
| **Leaky Bucket** | запросы «вытекают» с постоянной скоростью; сглаживает поток. |
| **Fixed Window** | счётчик за фиксированное окно (минуту); просто, но всплеск на границе окон. |
| **Sliding Window** | скользящее окно; точнее, без всплесков на границах. |

```php
<?php
// Token Bucket на Redis (упрощённо)
class RateLimiter
{
    public function __construct(
        private Redis $redis,
        private int $capacity = 100,     // макс. токенов
        private float $refillRate = 10,  // токенов в секунду
    ) {}

    public function allow(string $clientId): bool
    {
        $key = "rate:$clientId";
        $now = microtime(true);

        $tokens = (float) ($this->redis->hGet($key, 'tokens') ?: $this->capacity);
        $last   = (float) ($this->redis->hGet($key, 'ts') ?: $now);

        // Пополнить пропорционально прошедшему времени
        $tokens = min($this->capacity, $tokens + ($now - $last) * $this->refillRate);

        if ($tokens < 1) {
            return false;                 // лимит исчерпан → HTTP 429
        }

        $this->redis->hMSet($key, ['tokens' => $tokens - 1, 'ts' => $now]);
        return true;
    }
}
```

Ответ при превышении — **HTTP 429 Too Many Requests** с заголовками `Retry-After`, `X-RateLimit-Remaining`. Часто реализуется на уровне API Gateway. В PHP есть `symfony/rate-limiter`.

### Idempotency

**Проблема:** при повторах (retry) и at-least-once доставке сообщений одна операция может выполниться дважды — двойное списание денег, дубль заказа.

**Решение:** операция должна давать **одинаковый результат при повторном выполнении**. GET/PUT/DELETE идемпотентны по природе; POST — нет, его делают идемпотентным через **ключ идемпотентности**.

```php
<?php
class PaymentController
{
    public function charge(Request $request): Response
    {
        // Клиент присылает уникальный ключ (напр. UUID) для каждой логической операции
        $key = $request->getHeader('Idempotency-Key');

        // Уже обрабатывали этот ключ? — вернуть сохранённый результат
        if ($cached = $this->store->get($key)) {
            return $cached;               // повтор не выполняет списание заново
        }

        $result = $this->paymentService->charge(
            amount: $request->get('amount')
        );

        $this->store->set($key, $result, ttl: 86400);  // сохранить результат под ключом
        return $result;
    }
}
```

Способы обеспечения идемпотентности:
- **Idempotency-Key** + хранилище обработанных ключей (для API);
- **дедупликация по message ID** у консьюмеров очередей;
- **уникальные ограничения в БД** (`UNIQUE` на бизнес-ключ);
- **условные обновления** (optimistic locking, версии).

> Идемпотентность — обязательное условие безопасных повторов и работы с очередями (которые почти всегда at-least-once).

## Данные и транзакции

В микросервисах у каждого сервиса **своя БД** (Database per Service). Это исключает обычные ACID-транзакции между сервисами — нужны другие подходы.

### Saga

**Проблема:** бизнес-операция охватывает несколько сервисов (заказ → оплата → резерв склада → доставка). Распределённой ACID-транзакции нет; двухфазный коммит (2PC) плохо масштабируется и блокирует.

**Решение:** Saga — последовательность **локальных транзакций**. Каждый шаг публикует событие/команду для следующего. При сбое выполняются **компенсирующие транзакции** (откатывающие действия) в обратном порядке.

```
Создать заказ ─▶ Списать оплату ─▶ Зарезервировать товар ─▶ Создать доставку
                                          │ (нет на складе)
                                          ▼
            Вернуть оплату ◀── Отменить заказ      ← компенсации
```

Два стиля координации:

| | Хореография (Choreography) | Оркестрация (Orchestration) |
|---|----------------------------|------------------------------|
| Управление | сервисы реагируют на события друг друга | центральный оркестратор раздаёт команды |
| Связанность | децентрализовано | централизовано |
| Плюсы | нет единой точки, проще для коротких саг | явная логика, проще отлаживать длинные саги |
| Минусы | сложно отследить ход, цикличные зависимости | оркестратор — узкое место/сложность |

```php
<?php
// Оркестратор (упрощённо)
class OrderSaga
{
    public function handle(CreateOrder $cmd): void
    {
        $steps = [];
        try {
            $order = $this->orders->create($cmd);
            $steps[] = fn() => $this->orders->cancel($order->id);   // компенсация

            $payment = $this->payments->charge($order);
            $steps[] = fn() => $this->payments->refund($payment->id);

            $this->inventory->reserve($order);                       // может упасть
            // ... успех
        } catch (\Throwable $e) {
            // Откатываем выполненные шаги в обратном порядке
            foreach (array_reverse($steps) as $compensate) {
                $compensate();
            }
            throw $e;
        }
    }
}
```

> Saga даёт **eventual consistency**, а не мгновенную. Компенсация ≠ откат БД: это новое действие (вернуть деньги), которое тоже может упасть — поэтому шаги делают идемпотентными и логируют.

### Transaction Outbox

**Проблема:** сервис должен атомарно (1) сохранить данные в свою БД и (2) опубликовать событие в брокер. Но БД и брокер — разные системы; нет общей транзакции. Если упасть между ними — данные сохранены, а событие потеряно (или наоборот).

**Решение:** записывать событие в **таблицу outbox в той же БД-транзакции**, что и бизнес-данные. Отдельный процесс (relay/poller или CDC) читает outbox и публикует события в брокер, помечая отправленные.

```php
<?php
// 1) Бизнес-данные и событие — в ОДНОЙ транзакции БД
$pdo->beginTransaction();
try {
    $pdo->prepare('INSERT INTO orders (id, total) VALUES (?, ?)')
        ->execute([$orderId, $total]);

    // Событие в ту же БД — атомарно с заказом
    $pdo->prepare('INSERT INTO outbox (id, type, payload, created_at) VALUES (?, ?, ?, ?)')
        ->execute([uuid(), 'OrderPlaced', json_encode($payload), date('c')]);

    $pdo->commit();
} catch (\Throwable $e) {
    $pdo->rollBack();
    throw $e;
}
```

```php
<?php
// 2) Отдельный воркер публикует и помечает отправленные (Message Relay)
foreach ($outbox->fetchUnpublished(limit: 100) as $event) {
    $broker->publish($event->type, $event->payload);
    $outbox->markPublished($event->id);
}
```

Способы чтения outbox:
- **Polling Publisher** — воркер периодически опрашивает таблицу;
- **CDC (Change Data Capture)** — чтение лога БД (Debezium + Kafka) без нагрузки запросами.

> Гарантирует **at-least-once** доставку (событие может уйти дважды при сбое после publish, до markPublished) → потребители должны быть идемпотентны. Парный паттерн на стороне приёма — **Inbox** (дедупликация входящих).

### CQRS

**CQRS (Command Query Responsibility Segregation)** — разделение моделей для **записи** (команды, меняющие состояние) и **чтения** (запросы).

**Проблема:** одна модель, оптимальная и для сложной бизнес-логики записи, и для разнообразных быстрых чтений — компромисс, который плохо масштабируется (особенно при чтении >> записи).

**Решение:** две модели.
- **Command side** — нормализованная, с валидацией и бизнес-правилами; принимает команды.
- **Query side** — денормализованные read-модели (проекции), заточенные под конкретные запросы/экраны; обновляются асинхронно по событиям.

```
Команды ─▶ [Write Model] ─▶ событие ─▶ [Read Model(s)] ◀─ Запросы
           (нормализ. БД)              (денормализ. проекции)
```

```php
<?php
// Команда — меняет состояние, ничего не возвращает (кроме подтверждения)
final class CreateOrder { public function __construct(public int $userId, public array $items) {} }

class CreateOrderHandler {
    public function __invoke(CreateOrder $cmd): void { /* валидация, запись, событие */ }
}

// Запрос — только читает, не меняет состояние
final class GetOrderSummary { public function __construct(public int $orderId) {} }

class GetOrderSummaryHandler {
    public function __invoke(GetOrderSummary $q): OrderSummaryView {
        // Чтение из денормализованной проекции — быстро, без JOIN'ов
        return $this->readDb->findSummary($q->orderId);
    }
}
```

Плюсы: независимое масштабирование чтения/записи, оптимальные модели под задачу.
Минусы: сложность, eventual consistency между моделями, дублирование данных.

> CQRS **не** требует разных БД — это разделение моделей. Часто (но не всегда) сочетается с Event Sourcing и EDA. Применять только там, где выгода перевешивает сложность (высокая нагрузка на чтение, сложный домен).

---

### Event Sourcing

**Проблема:** обычная модель хранит только **текущее** состояние; история изменений теряется (или ведётся костыльно через audit-логи). Трудно ответить «как пришли к этому состоянию», воспроизвести баг, построить новые проекции из прошлого.

**Решение:** хранить не текущее состояние, а **полную последовательность событий**, которые к нему привели. Текущее состояние = свёртка (replay) всех событий.

```
Состояние НЕ хранится напрямую. Хранится журнал:
  #1 OrderCreated(items=[A,B])
  #2 ItemAdded(C)
  #3 ItemRemoved(A)
  #4 OrderPaid(amount=100)
→ Текущее состояние = применить события по порядку
```

```php
<?php
class Order
{
    private array $items = [];
    private bool $paid = false;
    private array $pendingEvents = [];

    // Восстановление из истории (replay)
    public static function reconstitute(array $events): self
    {
        $order = new self();
        foreach ($events as $event) {
            $order->apply($event);        // НЕ записывает новые события
        }
        return $order;
    }

    public function addItem(string $sku): void
    {
        $event = new ItemAdded($sku);
        $this->apply($event);
        $this->pendingEvents[] = $event;  // к сохранению в event store
    }

    private function apply(object $event): void
    {
        match (true) {
            $event instanceof ItemAdded   => $this->items[] = $event->sku,
            $event instanceof ItemRemoved => $this->items = array_diff($this->items, [$event->sku]),
            $event instanceof OrderPaid   => $this->paid = true,
            default => null,
        };
    }
}
```

Плюсы: полный аудит «из коробки», возможность построить любую проекцию задним числом, temporal queries («состояние на дату»), отладка через replay.
Минусы: высокая сложность, эволюция схемы событий (versioning/upcasting), быстрый рост журнала → нужны **снапшоты** (периодически сохранять состояние, чтобы не реплеить всё).

> Event Sourcing естественно сочетается с CQRS: read-модели строятся как проекции потока событий. Это мощный, но «дорогой» паттерн — не применять без явной потребности в истории/аудите.

## Миграция

### Strangler Fig

**Проблема:** есть большой легаси-монолит, который нужно перевести на микросервисы. Переписать всё разом («big bang rewrite») — долго, рискованно и часто проваливается: бизнес замораживается, а новая система не успевает за изменениями старой.

**Решение:** постепенно «обвивать» и заменять монолит сервисами, кусок за куском, пока старая система не отомрёт. Название — от фикуса-душителя (*strangler fig*), который прорастает вокруг дерева-хозяина и со временем полностью его замещает.

Как это работает: перед монолитом ставится **фасад/прокси** (часто роль играет API Gateway), который маршрутизирует запросы. Сначала всё идёт в монолит. Затем функциональность по одной выносится в новые сервисы, и прокси постепенно перенаправляет соответствующие маршруты на них.

```
Этап 1: всё в монолите
   Клиент ─▶ [Proxy/Facade] ─▶ Монолит (всё)

Этап 2: часть вынесена
   Клиент ─▶ [Proxy/Facade] ─┬─▶ Payment Service   (новое)
                             ├─▶ Order Service      (новое)
                             └─▶ Монолит            (остаток)

Этап 3: монолит «задушен» и удалён
   Клиент ─▶ [Proxy/Facade] ─┬─▶ Payment Service
                             ├─▶ Order Service
                             └─▶ Catalog Service ...
```

```php
<?php
// Фасад маршрутизирует: вынесенные пути → новый сервис, остальное → монолит
class StranglerFacade
{
    // Маршруты, уже мигрированные в микросервисы
    private array $migrated = [
        '#^/api/payments#' => 'http://payment-service',
        '#^/api/orders#'   => 'http://order-service',
    ];

    public function route(Request $request): Response
    {
        $path = $request->getPathInfo();

        foreach ($this->migrated as $pattern => $serviceUrl) {
            if (preg_match($pattern, $path)) {
                return $this->proxyTo($serviceUrl, $request);   // новый сервис
            }
        }

        return $this->proxyTo('http://legacy-monolith', $request); // ещё в монолите
    }
}
```

Порядок и приёмы миграции:
- начинают с **наименее связанных** и **наиболее ценных** кусков (быстрый эффект, низкий риск);
- на время перехода данные могут синхронизироваться между монолитом и сервисом (CDC, события, общая БД как временный компромисс);
- каждый шаг **обратим** — если новый сервис проблемный, прокси возвращает трафик в монолит;
- сопутствующий паттерн — **Anti-Corruption Layer (ACL)**: прослойка, изолирующая чистую модель нового сервиса от «грязных» структур легаси.

Плюсы: инкрементально и низкорисково; ценность поставляется постепенно; в любой момент можно остановиться. Минусы: долгое сосуществование двух систем; нужен надёжный фасад; временная сложность синхронизации данных.

> Strangler Fig — это паттерн **процесса миграции**, а не рантайма. Альтернатива ему — рискованный «big bang». Прокси-фасад почти всегда реализуется на базе **API Gateway**.

## Как паттерны сочетаются

Паттерны редко применяются поодиночке — они дополняют друг друга:

- **Устойчивый вызов:** `Timeout` → `Retry с jitter` → `Circuit Breaker` → `Fallback`. Каждый уровень страхует следующий.
- **Надёжные события:** бизнес-запись + `Transaction Outbox` → брокер → потребитель с `Idempotency` (Inbox).
- **Распределённая операция:** `Saga` поверх `Event-Driven` + `Outbox` для надёжной публикации шагов.
- **Чтение/запись под нагрузкой:** `CQRS` + (опц.) `Event Sourcing` + проекции через `EDA`.
- **На входе в систему:** `API Gateway` с `Rate Limiting` и auth.

```
Клиент ─▶ API Gateway (rate limit, auth)
            └─▶ Order Service
                  ├─ запись + Outbox ──▶ брокер ──▶ другие сервисы (идемпотентно)
                  └─ sync-вызов Payment: Timeout+Retry+CircuitBreaker+Fallback
                       └─ распределённая операция = Saga с компенсациями
```

## Шпаргалка для собеседования

| Паттерн | Решает | Суть |
|---------|--------|------|
| **API Gateway** | единая точка входа | маршрутизация + сквозные задачи (auth, rate limit) |
| **Sync messaging** | нужен ответ сейчас | HTTP/gRPC, блокирующий вызов |
| **Async messaging** | развязка сервисов | очереди, «отправил и забыл» |
| **Event-Driven** | слабая связанность | сервисы реагируют на события, не зная друг о друге |
| **Timeouts & Retries+jitter** | временные сбои | ограничить ожидание + повтор с разбросом |
| **Circuit Breaker** | каскадные отказы | размыкать цепь к больному сервису |
| **Fallback** | graceful degradation | запасной ответ при сбое |
| **Rate Limiting** | перегрузка/абьюз | лимит запросов (token/leaky bucket) → 429 |
| **Idempotency** | дубли при повторах | одинаковый результат при повторе (ключ идемпотентности) |
| **Saga** | транзакция через сервисы | цепочка локальных транзакций + компенсации |
| **Transaction Outbox** | атомарность БД+событие | событие в ту же БД-транзакцию, релей публикует |
| **CQRS** | чтение ≫ запись | разные модели для команд и запросов |
| **Event Sourcing** | аудит/история | хранить события, а не текущее состояние |
| **Strangler Fig** | миграция монолита | постепенно заменять монолит сервисами через прокси-фасад |

**Частые вопросы:**
- *Почему нельзя 2PC между сервисами?* — блокирует ресурсы, плохо масштабируется, единая точка отказа координатора → Saga.
- *Чем компенсация в Saga отличается от rollback?* — это новое бизнес-действие (вернуть деньги), а не откат БД; может само упасть.
- *Зачем jitter в ретраях?* — чтобы клиенты не повторяли синхронно и не создавали retry storm.
- *Почему нужна идемпотентность?* — очереди и ретраи дают at-least-once, операция может выполниться дважды.
- *Outbox гарантирует exactly-once?* — нет, at-least-once; exactly-once достигается идемпотентностью потребителя.
- *Circuit Breaker состояния?* — Closed → Open → Half-Open.
- *CQRS обязательно с Event Sourcing?* — нет, это независимые паттерны (часто сочетаются).
- *Чем Strangler Fig лучше переписывания с нуля?* — инкрементально и низкорисково: каждый шаг обратим, ценность поставляется постепенно, нет «заморозки» бизнеса как при big bang.
